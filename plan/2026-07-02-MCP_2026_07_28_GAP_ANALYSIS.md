# MCP 2026-07-28 Spec — Gap Analysis for LocalRouter

**Status:** analysis only, no implementation yet.
**Spec timeline:** RC locked 2026-05-21; final publishes **2026-07-28**. Tier-1 SDK
betas (Python/TS/Go/C#) already ship RC support, so new-revision clients and
servers will appear in the wild shortly after publication.
**LocalRouter today:** custom Rust implementation pinned to `2025-11-25`
(`crates/lr-mcp/src/protocol.rs:158`), no official SDK, acting as both MCP
server/gateway (downstream clients → `/mcp/*`) and MCP client/bridge (backend
servers over stdio/streamable-HTTP/websocket).

## Why nothing breaks on day one

New-revision clients are required to interoperate with older servers (the
ecosystem is full of them): they probe via `server/discover` / the
`MCP-Protocol-Version` header and fall back to the `initialize` handshake.
Old clients keep working against LocalRouter unchanged. The risk is gradual:
as flagship clients (Claude Desktop etc.) adopt 2026-07-28 natively, LocalRouter
increasingly runs on their legacy-compat path, and new-only peers (and new
features: Tasks, MCP Apps, extensions) won't work at all.

## Tier 0 — security fixes, do immediately (independent of spec adoption)

1. **RFC 9207 `iss` validation (SEP-2468, MUST).** Our OAuth callback validates
   only the CSRF `state` param (`crates/lr-oauth/src/browser/callback_server.rs:45,89`,
   `flow_manager.rs:73`); the `iss` parameter on authorization responses is never
   checked → mix-up attack exposure. Small, contained fix.
2. **Key OAuth credentials by issuer (SEP-2352, MUST).** Tokens/secrets are
   stored in the keyring keyed by `server_id` (`crates/lr-mcp/src/oauth.rs:689,835,1017`),
   not by the authorization server's issuer. If a backend's auth server changes,
   we'd replay credentials to the wrong issuer. Store issuer alongside and
   refuse reuse on mismatch; re-register/re-auth when the issuer changes.

## Tier 1 — core 2026-07-28 support (target: around spec publication)

Dual-revision support; keep the 2025-11-25 path for old peers (12-month
deprecation windows make removal a 2027 concern).

3. **Stateless lifecycle as server (SEP-2575).** Gateway must accept requests
   with no prior `initialize`: read `_meta` keys
   `io.modelcontextprotocol/{protocolVersion,clientInfo,clientCapabilities}`
   on every request, and implement **`server/discover`** (MUST for servers).
   Today the gateway never even inspects the client's `protocolVersion`
   (`gateway.rs:1712` parses capabilities only), and a request without
   `initialize` hits a session with no backend transports and fails.
4. **Session redesign (SEP-2567).** `Mcp-Session-Id` is gone; list endpoints
   must not vary per *connection*. Good news: our variation is per
   *authenticated client identity* (API key), which remains legal — it maps to
   `cacheScope: "private"`. Work: state keyed by client identity instead of
   connection UUID; per-session backend transports
   (`transport/session_transport_set.rs`, plan 2026-03-26) become per-client.
5. **`subscriptions/listen` (SEP-2575).** Replaces the GET SSE endpoint and
   `resources/subscribe`/`unsubscribe`. Our downstream GET `/mcp` +
   `endpoint` event (`routes/mcp.rs:85-208`) is the *old HTTP+SSE transport*,
   now formally Deprecated. Implement the long-lived POST stream with typed
   opt-ins (`toolsListChanged`, …) and `io.modelcontextprotocol/subscriptionId`
   tagging; keep old SSE for legacy clients.
6. **MRTR (SEP-2322).** Server-initiated `sampling/createMessage`,
   `elicitation/create`, `roots/list` are replaced by returning
   `InputRequiredResult` (`resultType: "input_required"`, `inputRequests`,
   opaque `requestState`) and having the client retry with `inputResponses`.
   Affects: elicitation manager (`gateway/elicitation.rs`), sampling
   passthrough (`routes/mcp.rs:651,751`), roots handling. Gateway must
   translate in both directions (new client ↔ old backend, old client ↔ new
   backend). Also: **every result needs `resultType: "complete"`** when
   speaking 2026-07-28; results from old backends missing it are treated as
   complete. Note the 2025-11-25 `notifications/elicitation/complete` and URL
   elicitation `elicitationId` are removed in the new revision.
7. **Transport headers (SEP-2243).** Streamable HTTP POSTs require `Mcp-Method`
   + `Mcp-Name`; servers must reject header/body mismatches
   (`HeaderMismatchError`, now `-32020`). Neither is sent by our client
   transport (`transport/sse.rs`) nor read by our server (`routes/mcp.rs`).
   Also send/read `MCP-Protocol-Version`.
8. **Caching fields (SEP-2549).** `ttlMs` + `cacheScope` required on results of
   `tools/list`, `prompts/list`, `resources/list`, `resources/read`,
   `resources/templates/list`. Merger (`gateway/merger.rs`) must emit them
   (`cacheScope: "private"` given per-client filtering) and ideally use
   backend-provided TTLs to reduce backend polling. Tools SHOULD be returned
   in deterministic order — check merger ordering.
9. **Removed methods for new peers.** `ping`, `logging/setLevel`,
   `notifications/roots/list_changed` are gone in 2026-07-28 (`gateway.rs:1438`,
   `router.rs:168` handle ping/logging today). Log level moves to per-request
   `_meta` `io.modelcontextprotocol/logLevel`. Keep for old peers.
10. **Error codes.** `-32002` → `-32602` for missing resources when speaking
    2026-07-28 (`protocol.rs:162`, `gateway_resources.rs:269,331,344`); add the
    renumbered `-32020..-32022` errors (HeaderMismatch,
    MissingRequiredClientCapability, UnsupportedProtocolVersion).
11. **Version bridging as a bridge/client.** For each backend: probe
    `server/discover` (also usable on stdio), fall back to `initialize` for old
    servers; track per-backend negotiated revision; translate lifecycle,
    MRTR, subscriptions, and error codes across revisions. This is genuinely a
    product opportunity: LocalRouter can advertise "new clients ↔ old servers
    and vice versa" bridging.
12. **Capabilities `extensions` map.** Pass through / merge the new
    `extensions` field on client/server capabilities (SEP-2133).
13. **SSE resumability removed.** We never implemented `Last-Event-ID`, so
    nothing to delete; ensure the client transport re-issues broken in-flight
    requests with a new request ID instead of expecting redelivery.
14. **Tool schema loosening (SEP-2106).** Schemas may now use full JSON Schema
    2020-12 (`$ref`, `oneOf`, …). We store schemas opaquely (good); verify the
    merger's description-stripping (`merger.rs:545-552,627-645`) tolerates
    schemas without top-level `properties`; never auto-dereference external
    `$ref` URIs (we don't).

## Tier 2 — strategic / optional

15. **Sampling deprecation (SEP-2577) — strategic upside.** Sampling, Roots,
    and Logging are Deprecated (functional ≥12 months). We have a large
    investment in sampling (gateway/sampling.rs, sampling_approval.rs,
    passthrough) — keep it for old peers, build nothing new on it. The spec's
    suggested replacement is "integrate directly with LLM provider APIs" —
    which is exactly what LocalRouter provides locally. Position LocalRouter's
    OpenAI-compatible endpoint as the migration path for servers that used
    sampling.
16. **Client ID Metadata Documents.** DCR (RFC 7591) is deprecated in favor of
    Client ID Metadata Documents. We never implemented DCR (statically
    configured client_id only) — leapfrog directly to CIMD for backend OAuth.
    The DCR `application_type` requirement (SEP-837) is moot for us.
17. **Tasks extension (`io.modelcontextprotocol/tasks`).** We never implemented
    experimental tasks, so no migration burden; adopting the extension could
    fit the coding-agents/long-running-work features later.
18. **MCP Apps (SEP-1865).** Sandboxed-iframe server UIs. Relevant to the
    desktop app long-term; watch, don't build.
19. **Trace context (SEP-414).** Propagate `traceparent`/`tracestate`/`baggage`
    `_meta` keys through the gateway — cheap, improves the observability story.

## Suggested sequencing

1. Tier 0 now (small, security-motivated, spec-version-independent).
2. Design doc for the dual-revision gateway core (items 3, 4, 11 are one
   architectural unit; 5–10, 12–14 hang off it) — this is the big lift given
   how session-centric `gateway.rs` (~3k lines) is today.
3. Implement behind per-peer revision detection; land before/around 2026-07-28.
4. Tier 2 opportunistically.
